# FastPPT 项目详细方案

## 1. 项目概述

### 1.1 项目目标

FastPPT 面向 A04 多模态 AI 互动式教学智能体赛题，聚焦高校教师备课场景，目标是将“资料理解、教学意图澄清、课件生成、讲义导出、页级修改”整合为一个连续闭环。

项目的核心追求不是“一次性自动出一套课件”，而是：

1. 先准确理解教师要讲什么。
2. 再把教师已有资料真正用于生成。
3. 最后生成一份可修改、可复用、可继续迭代的课件成果。

### 1.2 一句话定位

> FastPPT 是一个面向高校教师的教学设计执行系统，而不是普通主题驱动型 AI PPT 生成器。

## 2. 赛题理解与问题定义

## 2.1 我们对赛题的核心判断

结合赛题文档、工作区结论文档与教师反馈，当前可以明确：

1. 赛题核心不是“从零帮老师想一门课”，而是“把老师已有意图与资料稳定执行出来”。
2. 高价值竞争点不在于炫酷效果，而在于多模态资料利用、教学可用性、生成后修改和可追溯性。
3. 高校教师对 AI 最大的不满不是“生成得不够快”，而是“没听懂、乱猜、乱改、不好用”。

## 2.2 真实用户与主场景

### 目标用户

- 有真实授课任务的高校教师
- 已经具备教学判断和课程思路
- 手中已有教材、旧 PPT、讲义、论文、图片、视频等资料

### 主场景

当前最重要的真实场景是：

> 基于旧 PPT 或同类课程 PPT，在保留原有教学结构和风格的前提下，增加新案例、新数据、新政策和更相关的示意图，形成一版更适合当前授课的新课件。

### 次级场景

1. 教材 / PDF / Word 资料转课件
2. 教师脑中有明确思路，但表达尚未结构化
3. 将公开课或教学视频中的案例与讲法吸收进课件

## 2.3 用户痛点

### 1. 需求表达成本高

老师不愿意为了生成一份课件去写一大段高质量 prompt，更不接受模糊输入后由 AI 自行猜测。

### 2. 资料利用率低

现有教材、旧 PPT、PDF、图表和案例经常无法真正进入新课件，而只是被“参考一下”。

### 3. 结果不够可教

现有工具往往要么太花哨，不适合教学；要么太空、缺图、缺相关案例。

### 4. 修改成本高

真实工作流里，教师最常见的不是整套重做，而是：

- 只改某一页
- 增加一个案例
- 删除过于复杂的部分

如果系统一修改就带乱整套课件，实际使用价值就很低。

## 3. 产品定位与价值主张

## 3.1 定位

FastPPT 的产品定位可以概括为：

- `teacher-facing`
- `代劳执行`
- `结构化引导`
- `可修改`
- `可积累`

系统不替老师决定讲什么，而是将老师已经明确的教学设计，转化为机器可执行、可验证、可修订的过程。

## 3.2 价值优先级

根据工作区文档与教师反馈，FastPPT 的价值优先级已经明确为：

1. 真正理解教师意图
2. 支持稳定修改闭环
3. 把多模态资料真正搬进课件
4. 输出高质量可用结果

这里第 4 项不是起点，而是前 3 项落实后的自然结果。

## 4. 当前版本的产品约束

## 4.1 生成前的三项必确认

FastPPT 在当前版本中明确要求，生成前至少需要确认以下信息：

1. 教学目标
2. 面向学生类型
3. 重点难点

这三项已经在当前系统与文档中被确定为生成前放行条件。

## 4.2 结果“可用”的四项标准

系统产出是否可用，不以“生成成功”作为标准，而以以下四项作为评价门槛：

1. 内容正确
2. 结构顺
3. 像老师自己的课
4. 能回到给定资料，不乱编

这四条是当前方案中最重要的质量共识。

## 5. 当前系统方案

## 5.1 主链路

当前 FastPPT 的方案主线可以表述为：

`Teacher Input -> TeachingSpec -> SourceAnchor -> PageOutline -> SlideAST -> Render -> Verify -> Repair/Patch -> CourseMemory`

其中：

- 前半段解决“听不听得懂老师”
- 中间段解决“资料是否真正用上”
- 后半段解决“生成后能否稳定修改与长期复用”

这条主链既是产品设计主线，也是后续架构重构方向。

## 5.2 关键对象设计

### 1. TeachingSpec

用于把教师输入编译为结构化教学规格。当前代码中已存在相应对象与编译逻辑，能够稳定承载：

- topic
- teaching_goal
- audience
- duration
- style
- key_points
- difficulty_focus
- sources
- unresolved_fields

其作用是让系统状态不再依赖聊天记录和 prompt 拼接，而是依赖明确字段。

### 2. SourceAnchor

用于把教材、PDF、旧 PPT、图片、视频等资料转成可绑定、可定位、可追溯的证据锚点。

当前完整形态仍在推进，但它已经被明确为系统的一等对象，而不是普通 RAG chunk 的替代词。

### 3. PageOutline

用于把整份课件转换成逐页可解释的执行计划。它回答的是：

- 每一页承担什么教学功能
- 每一页讲什么知识点
- 每一页需要哪些素材与证据

该对象是后续两阶段生成和页级修改的关键。

### 4. SlideAST / blocks

用于把页面从自由文本压缩为结构化中间表示。当前系统已经开始推进“可插拔信息块”方向，后端已有 `slide_blocks` 协议，覆盖：

- TextBlock
- BulletBlock
- CodeBlock
- FormulaBlock
- CaseBlock
- TableBlock
- ImageBlock
- FlowchartBlock

这为后续的前端渲染、PPT 导出、局部 patch 打下了基础。

### 5. Verification / Repair / Patch

这是系统与普通生成器拉开差距的关键方向。当前已在文档中被明确为正式主链组成部分，目标是实现：

- 渲染后验证
- 低风险局部修补
- 页级和块级 patch

## 6. 当前已落地能力与演示基线

## 6.1 已落地能力

根据仓库文档、代码与运行状态，当前可确认已具备以下能力：

1. 多文件上传：支持 PDF、DOCX、PPTX、TXT、图片和视频。
2. 对话采集意图：收集 topic、goal、audience、difficulty、key points、duration、style。
3. `TeachingSpec` 编译与预检：后端可检查必填字段缺失情况。
4. 课件生成：输出 `slides_json`，供前端预览。
5. 真实导出：支持生成 `PPTX` 与 `DOCX`。
6. 页级修订：支持 revise 指令对指定页进行局部修改。
7. Demo 演示模式：当前后端运行于稳定 demo 模式。

## 6.2 已验证事实

当前可直接验证到的系统运行事实包括：

- 健康接口正常返回
- `demo_mode=true`
- `chat_mode=plain`
- `agent_enabled=false`
- `rag_mode=tfidf`
- `ppt_export=pptxgenjs+python-pptx-fallback`

这些事实的意义在于：FastPPT 现在不是停留在“概念方案”，而是已经具备可运行、可体验、可答辩演示的系统基础。

## 6.3 当前尚未完全完成的部分

为了保证方案表述真实可信，当前必须同时说明仍在推进的部分：

1. 真正的“先大纲、后逐页”的两阶段生成尚未完整打通。
2. 前端仍主要依赖 `page.type`，尚未全面切换到 `blocks[]` 渲染。
3. 公式 OMML 原生导出尚未完成。
4. 课程级长期记忆、验证闭环和自动修补仍属于下一阶段重点。

这一部分并不是弱点，相反，它体现了项目对现状与目标边界的清晰判断。

## 7. 技术方案说明

## 7.1 后端方案

- 技术栈：FastAPI + Python
- 负责多文件上传、意图处理、生成链路、导出、修订
- 当前代码入口集中在 `backend/main.py`、`backend/api/chat.py`、`backend/api/generate.py`

## 7.2 前端方案

- 技术栈：Vue 3 + Vite + Element Plus
- 负责对话输入、生成触发、预览、下载、修订交互
- 当前核心入口为 `PreviewPanel.vue` 等组件

## 7.3 生成与导出

当前导出链路采用：

- `slides_json` 作为前端预览中间层
- `PPTX` 作为课件交付物
- `DOCX` 作为教案交付物

系统当前显示的导出方式为：

- `pptxgenjs+python-pptx-fallback`

这保证了导出链路在 demo 模式下更稳定。

## 7.4 检索与资料利用

当前系统文档显示，检索层已有基础实现，当前运行模式下的 `rag_mode=tfidf` 表明系统已采用稳定优先的检索策略。下一步将继续向更强的资料锚点化与可追溯绑定演进。

## 8. 项目创新点

## 8.1 从“一键生成”转向“教学执行”

这是 FastPPT 与通用 AI PPT 工具的根本区别。系统的核心任务不是自由生成，而是围绕教师真实工作流进行执行。

## 8.2 用 TeachingSpec 承接教学意图

把需求从聊天记录提升为结构化合同，是当前方案最关键的系统性改进之一。

## 8.3 用 blocks 体系承接课件内容

把页面表达建模成可插拔内容块，为公式、代码、案例、图示、流程图等教学表达形式提供更稳定的结构基础。

## 8.4 强调页级 revise 与后续 patch

真实教师工作流中，修改是刚需而不是异常。FastPPT 已将页级修订纳入主链，这一点比“首次生成更快”更有价值。

## 8.5 保留课程级长期记忆方向

课程记忆不是一次性 RAG，而是课程资产的长期沉淀，这使系统具备长期复利潜力。

## 9. 演示方案

答辩中建议采用如下演示路径：

1. 上传旧 PPT 与教材片段
2. 输入教学目标、面向学生类型、重点难点
3. 触发生成，展示 slides 预览
4. 针对某一页提出修改要求
5. 展示 revise 后结果
6. 下载 PPTX 与 DOCX

这条演示路径的优点是：

- 符合教师真实工作流
- 能同时体现资料利用、结构化输入、生成能力和修改能力
- 比“给一个主题直接出 20 页”更有说服力

## 10. 风险与应对

## 10.1 风险一：把目标态写成已实现

### 应对

提交材料中严格区分：

- 已实现能力
- 已明确方向
- 下一阶段规划

## 10.2 风险二：生成结果偏题或不够可教

### 应对

- 生成前强制确认三项关键字段
- 强化资料绑定
- 以后续 verify/repair 作为稳定化方向

## 10.3 风险三：演示现场稳定性不足

### 应对

- 采用 demo 模式
- 使用 plain chat 路径，关闭非必要 agent 链路
- 现场优先展示已跑通的端到端流程

## 11. 路线图

## 11.1 P0：先把“可控首稿”做稳

重点包括：

1. TeachingSpec 继续强化
2. SourceAnchor 最小闭环
3. PageOutline 主链落地
4. `outline -> per-page generation` 两阶段生成打通
5. 前端支持 `blocks[]` 渲染

## 11.2 P1：再把“可用性”做强

重点包括：

1. 渲染后验证
2. 局部修补
3. 风格继承
4. 课程记忆
5. 图文相关性增强

## 11.3 P2：再做长期复利与扩展能力

重点包括：

1. patch engine
2. 更丰富的 block registry
3. 学科化表达能力
4. 更强的导出与长期课程工作区

## 12. 结论

FastPPT 的优势不在于“模型更大”或“生成更花”，而在于它对高校教师真实工作流的理解更准确：先理解、再绑定资料、再生成、再修改、再沉淀。当前系统已经具备多文件上传、结构化意图采集、PPTX/DOCX 导出和页级 revise 等可演示能力，同时保留了页级规划、验证修补、课程记忆等清晰的演进路线。

因此，FastPPT 既能回答评审“现在已经做到什么”，也能回答“为什么这个方向值得继续做”。这正是它作为比赛提交项目的核心说服力所在。
